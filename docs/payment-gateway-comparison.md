## 1. 후보 게이트웨이

### 1.1 포트원 (PortOne, 구 아임포트)

**한국 결제 통합 게이트웨이 — 카드사·간편결제·계좌이체·휴대폰결제 단일 API 통합.**

| 항목 | 평가 |
|---|---|
| 한국 결제 환경 | ◎ 모든 PG (KG이니시스, 토스페이먼츠, 나이스페이, 카카오페이, 토스페이, 네이버페이, 페이코) 단일 API |
| 글로벌 결제 | △ 일부 PG에서 해외카드 제한 지원, 본격 글로벌은 약함 |
| B2B 인보이스 | ◎ 전자세금계산서 자동 발행 연동 (한국 사업자 친화) |
| 개발자 경험 | ○ JS SDK + REST API. Next.js 통합 가이드 풍부. webhook 표준 |
| 구독 정기결제 | ○ V2 API 빌링키 모델 (간편결제 빌링키 포함) |
| 수수료 | 카드 약 3.0~3.4% + VAT, 간편결제 약 2.5% (계약 PG에 따름). 포트원 자체 수수료 없음 (PG 직결) |
| 환불 | ○ API 호출로 즉시. 청약철회 명시 가능 |
| Vercel 정합성 | ○ webhook URL 등록 가능. node 런타임 충분 |
| 한국 사업자 KYC | ◎ 사업자등록증으로 가입, 한국법인 정합 |
| 약점 | 글로벌 시장 약점 (USD 결제·해외 카드 비율 낮음). 국내 PG와 별도 계약 필요 (포트원이 PG 자체는 아님) |

### 1.2 Stripe

**글로벌 SaaS 표준. 한국 진출 후 KRW 직접 정산 가능.**

| 항목 | 평가 |
|---|---|
| 한국 결제 환경 | ○ 한국 카드 직접 결제 가능, KRW 정산. 단 카카오페이·토스페이 등 한국 간편결제 **미지원** |
| 글로벌 결제 | ◎ 135+ 통화, 모든 주요 카드, Apple/Google Pay |
| B2B 인보이스 | ○ Stripe Invoicing — 자동 인보이스. 단 한국 전자세금계산서는 **자체 연동 필요** (Stripe → 홈택스 별도 처리) |
| 개발자 경험 | ◎ 업계 표준. Next.js·Prisma 통합 예제 풍부, 테스트 모드 우수, 문서 최고 |
| 구독 정기결제 | ◎ Subscriptions 핵심 제품. 프로레이션·트라이얼·할인 코드 등 모두 지원 |
| 수수료 | 한국 카드 3.5% + ₩300 (해외 카드 +1%), 인보이스 0.4%, 정기결제 별도 수수료 없음 |
| 환불 | ◎ API/대시보드 모두. 부분환불 자유 |
| Vercel 정합성 | ◎ 공식 가이드, webhook 안정 |
| 한국 사업자 KYC | ○ 한국법인 가입 가능. 비즈니스 검증 후 KRW 정산 |
| 약점 | 한국 간편결제 부재 = **컨버전 손실 가능성 큼**. 한국 사용자 60~70%가 카카오페이·토스페이 선호 |

### 1.3 Polar

**개발자·크리에이터 친화 신생 결제. Stripe 위에 SaaS 빌링 추상화.**

| 항목 | 평가 |
|---|---|
| 한국 결제 환경 | △ 내부적으로 Stripe 사용 → 한국 간편결제 부재 동일 |
| 글로벌 결제 | ○ Stripe 위 빌더 — 글로벌 동일 |
| B2B 인보이스 | △ 자동 인보이스 가능, 한국 전자세금계산서 별도 |
| 개발자 경험 | ◎ TypeScript-first SDK, Next.js 통합 매우 단순, webhook DX 우수 |
| 구독 정기결제 | ○ Stripe 기반 |
| 수수료 | Polar 4% + Stripe 수수료 (즉 한국 카드 7.5%+). Merchant of Record 선택 시 부가세·sales tax 자동 처리 (편의성 vs 수수료 trade-off) |
| 환불 | ○ API |
| Vercel 정합성 | ◎ Vercel Marketplace 통합 |
| 한국 사업자 KYC | △ Polar 자체는 글로벌, 한국 법인 KYC는 가능하나 정산은 Stripe 거침 |
| 약점 | 수수료 가중. 한국 내 인지도 낮음. B2B 한국 인보이스 부담. **OriPics 한국 우선 시장에 부적합** |

### 1.4 Lemon Squeezy

**Merchant of Record 모델. Polar과 유사한 SaaS 빌링.**

| 항목 | 평가 |
|---|---|
| 한국 결제 환경 | △ 한국 간편결제 부재 |
| 글로벌 결제 | ◎ Merchant of Record로 sales tax/VAT 자동 처리 |
| B2B 인보이스 | △ 한국 전자세금계산서 별도 처리 필요 |
| 수수료 | 5% + $0.50 (단일, 정기결제 포함) |
| 약점 | Polar와 유사 — 한국 시장 약점 |

### 1.5 토스페이먼츠 (직계약 PG)

**포트원 대안 — 토스 직계약. 토스페이 + 카드 + 간편결제 직결.**

| 항목 | 평가 |
|---|---|
| 한국 결제 환경 | ◎ 카드·토스페이·계좌이체 등. 카카오페이·네이버페이는 별도 필요 |
| 글로벌 결제 | △ 해외 카드 일부 |
| B2B 인보이스 | ○ 전자세금계산서 연동 가능 |
| 개발자 경험 | ◎ 토스페이먼츠 SDK 매우 우수, 한국 PG 중 최고 평가 |
| 구독 정기결제 | ○ 빌링 API |
| 수수료 | 카드 약 2.8~3.0% (사업 규모 따라 협상) |
| 약점 | 단일 PG라 카카오페이·네이버페이는 별도 통합 필요 |

## 4. 추천 — 단계별 채택 전략

### Phase 1 (베타~출시 6개월) — **포트원 단독 채택**
**이유**:
- OriPics 초기 사용자는 한국 우세 (LinkedIn·트위터 한국 OSS 커뮤니티·교통사고 페르소나 모두 한국 기반)
- 카카오페이·토스페이·네이버페이 = 한국 사용자 컨버전 60~70% 좌우
- Pro ₩9,900 단가 × 한국 카드 수수료 3.2% = 건당 약 ₩317 → 마진 영향 작음
- 한국 사업자(SantaHades Co., Ltd.) 부가세·전자세금계산서 자동화 강점
- 한국 PG(KG이니시스/토스페이먼츠/나이스페이) 중 1개 + 간편결제 묶음 동시 가입 권장 (포트원 단일 API로 통합)

**예상 셋업**:
- 포트원 V2 SDK + REST API
- PG 계약: 토스페이먼츠 (수수료 + DX 균형)
- 간편결제: 카카오페이 + 토스페이 (베타 시작) → 출시 시 네이버페이 추가
- 빌링키 발급 → 매월 자동 결제
- 환불 정책: 청약철회 7일 + 결제일 기준 일할계산

### Phase 2 (출시 6~12개월, 글로벌 사용자 5%+ 시) — **Stripe 추가**
**조건**: 글로벌 사용자 비중 누적 5% 도달, USD 결제 요청 누적 N건 이상
**구현**: 결제 게이트웨이 추상화 레이어를 처음부터 도입 → 추가 시 Pro·Business 결제 라우터에 분기 추가

### Phase 3 (Business 본격) — Stripe Invoicing 또는 자체 청구서
한국 B2B는 포트원 인보이스, 글로벌 B2B는 Stripe Invoicing.

## 5. 추상화 레이어 설계 (Phase 1 시점에 도입 권장)

미래 Stripe 추가를 대비해 **PaymentGateway 인터페이스**를 처음부터 도입:

```typescript
// apps/web/src/lib/payment/gateway.ts
export interface PaymentGateway {
  createCustomer(user: User): Promise<{ customerId: string }>;
  createSubscription(opts: { customerId: string; planId: 'pro_monthly' | 'pro_yearly' | 'business' }): Promise<{ subscriptionId: string; status: string }>;
  cancelSubscription(subscriptionId: string): Promise<void>;
  refund(paymentId: string, amount?: number): Promise<void>;
  handleWebhook(req: NextRequest): Promise<{ event: WebhookEvent }>;
}

export const portoneGateway: PaymentGateway = { /* ... */ };
// 미래: export const stripeGateway: PaymentGateway = { /* ... */ };
```

이렇게 하면 Phase 2에서 Stripe 도입 시 라우터에서 사용자 지역 기반 분기만 추가하면 됨.
