## 2. 백엔드 크레딧 회계 (사용자 비노출)

**2026-05-17 갱신** — 인증(`confirm`) ↔ 공개링크 생성(`publish`) 2단계 분리. proof는 link 비용 미포함. CERTIFICATE_PDF는 Storage 캐시 도입.

| 액션 | 기본 크레딧 | 비고 |
|---|---|---|
| **검증 조회** (`/api/verify`) — **로그인 필수**, 호출자 계정에서 차감. 미공개 인증 이미지면 무차감 404 응답. | **−1** × multiplier | 본인 published 이미지는 면제 |
| **이미지 인증 1회** (Standard, F/C) — `/api/links/confirm` | **−3** × multiplier | proof만. Storage·DB 미접근 |
| **사진 인증 1회** (Verified, 모바일 P) — `/api/links/confirm` | **−4** × multiplier | proof만. Pro 한정 |
| **공개링크 생성 1회** — `/api/links/publish` | **−2** | 사이즈 무관 1× 고정. C2PA·Storage·DB row 생성을 모두 처리 |
| **증명서 PDF 발급 1회** — `/api/links/[id]/certificate` | **−10** | Pro 한정. 첫 발급 후 Storage 캐시 → 재다운로드 무료, `?reissue=1`로 재발급 −10 |

**Free 20크레딧 시 가능 횟수 (1× 기준)**:
- 인증만 (downloadable stamped PNG만) — Standard 6건 / Verified 5건
- 인증 + 공개링크 — Standard 4건 / Verified 3건 (각 인증 후 LINK_CREATE −2 추가)
- 검증만 (공개링크 가진 타인 이미지) — 20건


미공개 인증 단계에서는 서버 측 무상태 (Storage 0, DB 0) — cron 정리 불필요.

### 2.1 이미지 사이즈별 multiplier (2026-05-13 도입)

대형 이미지는 LSB 임베드 부하·저장·전송 비용이 비례 증가하므로 **사이즈 multiplier**를 적용한다. LINK_CREATE·CERTIFICATE_PDF는 사이즈 무관 → **1× 고정**.

| 사이즈 구분 | Multiplier | IMAGE_PROOF | VERIFIED_PROOF | VERIFY_QUERY |
|---|---|---|---|---|
| 긴 변 ≤ 1800px | **1×** | −3 | −4 | −1 |
| 긴 변 > 1800 AND 픽셀 ≤ 100MP | **2×** | −6 | −8 | −2 |
| 픽셀 > 100,000,000 (1억) | **3×** | −9 | −12 | −3 |

**적용 위치**:
- `/api/sign` + `/api/links/confirm` — 요청 width/height 기준
- `/api/verify` — 메타에 박힌 width/height 기준 (변조 불가)
- 클라이언트 측 사이즈 선택 UI (`/[locale]/page.tsx`) — 긴 변 > 1800px 이미지가 들어오면 "기준 사이즈" / "원본 사이즈" 체크박스 노출, 두 옵션 모두 선택 시 각각 별도 결과물 생성·각각 차감
- verify 확인 모달 — 동적 비용 표시 (`cost_notice_dynamic`)

**Free 접근성**: Multiplier 정책은 Free에도 동일 적용 (티어 게이트 없음). 50MP iPhone 셀카 1건(Standard, 2×) = 6크레딧 → Free 한 달치 20크레딧의 30%. 자연스러운 Pro 전환 트리거.

구현: [lib/credits/sizeMultiplier.ts](../src/lib/credits/sizeMultiplier.ts), 테스트: [sizeMultiplier.test.ts](../src/lib/credits/sizeMultiplier.test.ts).

**검증 조회 정책 (2026-05-11 갱신)**: 외부 검증 가능성은 contentcredentials.org·Adobe Photoshop·Truepic Verify 등 C2PA 외부 도구로 유지. OriPics 사이트 내 verify는 로그인 필수 + 차감.

**클라이언트 측 무료 detect (2026-05-11)**: 메인 페이지에서 PNG 업로드 시 `detectStamp()` (magic byte만 확인, 해시·서버 호출 X)로 인증 여부 무료 판단. 사용자가 "자세히 확인" 선택 시에만 풀 verify(`/api/verify`) 호출되어 -1 차감.

**용도**:
- 어뷰즈 차단 (이상 사용 패턴 탐지)
- B2B 인보이스에서 사용량 정확 보고
- 미래 API 상품(개발자용 별도 SKU) 추가 시 동일 인프라 재사용

## 4. 비로그인 정책

**모든 인증 액션은 회원가입 필수**. 비로그인 상태에서는 인증·공개링크 생성·조회 모두 차단.

### 채택 이유
1. C2PA + JPEG Trust 호환 마케팅이 충분한 신뢰 시그널 → 체험 단계 없이도 가입 동기 확보
2. 7일 보관 정책상 비로그인 결과는 휘발성 → "왜 7일 후 사라졌어요?" CS 부담 회피

4. 저작권·법적 효력 마케팅 시 익명 사용자 데이터 자체가 책임 부담

## 5. 갱신·만료·다운그레이드 정책

### 5.1 크레딧 갱신
- **갱신 단위**: 가입일(=결제일) 기준 매월
- **이월**: 불가. 매월 리셋 (cap 모델)
- **이유**: Stripe/포트원 결제 주기와 자연 정합. 사용자별 결제일 분산되어 시스템 부하 평탄화. 미사용 누적 부담 회피.

### 5.2 영구 보관 자료의 다운그레이드 처리
| 시나리오 | 정책 |
|---|---|
| Pro 활성 중 | ∞ 영구 보관 |
| Pro → Free 다운그레이드 직후 | **30일 grace period** (영구 보관 유지) |
| 30일 grace 만료 후 | 7일 보관 정책으로 회귀 → 가입자가 다시 Pro 결제하지 않으면 7일 후 자동 삭제 |
| 다운그레이드 후 30일 내 재결제 | grace 즉시 해제, 영구 보관 복원 |

### 5.3 사용자 알림
- 다운그레이드 시 → 즉시 이메일 + 앱 알림: "30일 후 영구 보관이 종료됩니다. 계속 보관하려면 Pro 재결제 또는 자료 다운로드"
- grace 마지막 7일/3일/1일 → 추가 알림
